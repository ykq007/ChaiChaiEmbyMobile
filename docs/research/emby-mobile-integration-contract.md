# Emby integration contract for the Mobile Client

Research date: 2026-07-15
Scope: official Emby contracts needed for interactive mobile browsing and playback
Conclusion: build a user-authenticated, server-scoped adapter that negotiates every playback from a truthful device profile, treats returned media-source/session data as authoritative, and gates optional behavior by the connected server rather than by assumptions recovered from the TV APK.

## Evidence boundary

This report uses Emby's official developer documentation and generated REST reference as the protocol authority. Emby says the REST API is available below `http[s]://host:port/emby/{apipath}`, supports JSON and XML, and exposes an interactive API Browser from a running server; that server-local browser is the final authority for the exact version under test ([REST API documentation](https://dev.emby.media/doc/restapi/index.html)).

The earlier [TV APK report](tv-apk-behavior-and-contracts.md) is used only to identify the client operations requiring confirmation. Its recovered Retrofit routes and DTOs align with the official surface, but decompiled code is not an upstream compatibility guarantee.

## Connection and authentication

The minimum connection flow is manual server URL entry, normalization to one server base, an unauthenticated `GET /System/Info/Public` probe, then user authentication. Emby also defines LAN discovery by UDP broadcast to port 7359 with the payload `who is EmbyServer?`; the response contains `Address`, `Id`, and `Name`, so discovery can supplement but must not replace manual entry ([Locating the Server](https://dev.emby.media/doc/restapi/Locating-the-Server.html), [SystemService reference](https://dev.emby.media/reference/RestAPI/SystemService.html)).

Every request identifies the client with an authorization header containing `Client`, `Device`, stable `DeviceId`, and `Version`, plus `UserId` after login. Authentication is `POST /Users/AuthenticateByName`; a successful response supplies `AccessToken`, which subsequent requests send as `X-Emby-Token`. A 401 generally means the token was revoked and requires reauthentication; explicit logout is `POST /Sessions/Logout` and revokes the token ([User Authentication](https://dev.emby.media/doc/restapi/User-Authentication.html)).

Tokens are scoped to a server identity: Emby explicitly instructs multi-server clients to save the token with the server `Id` returned by `/System/Info`, to avoid sending a token to the wrong server. Therefore the Mobile Client's credential key is `(ServerId, UserId)`, never a process-global token, and secrets must not be copied into logs, analytics, image cache keys, or user-visible URLs ([User Authentication](https://dev.emby.media/doc/restapi/User-Authentication.html), [getSystemInfo](https://dev.emby.media/reference/RestAPI/SystemService/getSystemInfo.html)).

Static API keys are for service/integration scenarios not tied to a user, whereas user authentication is the documented choice for interactively operated clients. The Mobile Client should not request or accept an administrator API key as its normal sign-in mechanism ([API-Key Authentication](https://dev.emby.media/doc/restapi/API-Key-Authentication.html), [REST API documentation](https://dev.emby.media/doc/restapi/index.html)).

Emby Connect is optional and should follow multi-server switching, not block the first contract. Emby recommends manual multi-server support first; Connect's mobile flow authenticates with the Connect service, lists servers, then exchanges each server's `AccessKey` at `/Connect/Exchange` for that server's local `UserId` and `AccessToken` ([Emby Connect](https://dev.emby.media/doc/restapi/Emby-Connect.html)).

## Library, search, and item contract

The browsing primitive is the authenticated user-scoped item API. A single `GET /Users/{UserId}/Items/{Id}` returns a full item, while list queries intentionally return reduced DTOs; callers must request required additions through the comma-separated `Fields` parameter. Paging, `ParentId`, `Recursive`, `IncludeItemTypes`, filters, and sorting should remain adapter parameters rather than screen-specific URL construction ([Browsing the Library](https://dev.emby.media/doc/restapi/Browsing-the-Library.html), [Item Information](https://dev.emby.media/doc/restapi/Item-Information.html)).

Home/library support should cover user views plus query-backed rails. Officially documented examples establish `/Users/{UserId}/Items/Latest` for latest content and `/Users/{UserId}/Items` with `Filters=IsResumable`, recursive item-type queries, and paging/sorting; latest results can group episodes by parent with `GroupItems=true`, so clients must not assume every returned card is the requested leaf type ([Latest Items](https://dev.emby.media/doc/restapi/Latest-Items.html), [Browsing the Library](https://dev.emby.media/doc/restapi/Browsing-the-Library.html), [getUsersByUseridItemsLatest](https://dev.emby.media/reference/RestAPI/UserLibraryService/getUsersByUseridItemsLatest.html)).

Item behavior is property/capability driven. `IsFolder` controls whether navigation opens a container, `MediaType` is preferred over exact `Type` for broad behavior, `MediaSources` supersedes item-level `MediaStreams` for playback decisions, and `UserData` carries user play/favorite state. Emby warns that plugins may add types, so unknown item types and absent optional properties must degrade safely ([Item Information](https://dev.emby.media/doc/restapi/Item-Information.html), [Item Types](https://dev.emby.media/doc/restapi/Item-Types.html)).

The static REST reference includes dedicated service operations for similar items and search, but generated definitions are broad and vary with server releases. The adapter should expose search as paged search results/hints and resolve a selected result to a full user-scoped item before rendering detail or playback; the exact search route and parameters must be locked against the target server's built-in API Browser ([REST API reference](https://dev.emby.media/reference/index.html), [getItemsByIdSimilar](https://dev.emby.media/reference/RestAPI/LibraryService/getItemsByIdSimilar.html?q=similar)).

## Images and caching

Fetch an image only when the DTO advertises it through image tags/counts; otherwise the endpoint can return 404. Item images use `/Items/{Id}/Images/{Type}` and indexed forms for multi-image types such as backdrops, screenshots, and chapters. The client should request bounded `Width`/`Height` or `MaxWidth`/`MaxHeight`, preserve primary-image aspect ratio when appropriate, and support parent image inheritance using the `Parent*ItemId` and `Parent*ImageTag(s)` properties ([Images](https://dev.emby.media/doc/restapi/Images.html)).

An image `Tag` is also a cache-version value. Including it in the URL yields strong cache headers and a changed image produces a changed URL; Emby recommends caching by the path after host/port so the same asset can survive LAN/remote address switching. Cache identity therefore needs server identity plus relative tagged path, while authentication is attached at request time rather than persisted in the cache key ([Images](https://dev.emby.media/doc/restapi/Images.html)).

## Playback negotiation

For every play attempt, call authenticated `POST /Items/{Id}/PlaybackInfo` with `UserId`, start position, selected media source/audio/subtitle indices when known, bitrate/channel ceilings, direct-play/direct-stream/transcode flags, and a `DeviceProfile`. The profile declares supported direct-play, transcoding, container, codec, response, and subtitle capabilities. The response returns `MediaSources`, `PlaySessionId`, and any `ErrorCode`; a no-compatible-stream or not-allowed result must become a user-facing playback failure, not a guessed fallback URL ([postItemsByIdPlaybackinfo](https://dev.emby.media/reference/RestAPI/MediaInfoService/postItemsByIdPlaybackinfo.html)).

The device profile must describe what the actual mobile player can decode and render on that device. Advertising unsupported codecs, containers, HDR profiles, subtitle delivery methods, channel counts, or bitrate ceilings can make the server choose an unusable route. Conversely, a conservative profile can force avoidable transcoding; profile construction therefore belongs behind a testable capability provider, not a hard-coded copy of the TV Client's observed 40 Mbps profile ([postItemsByIdPlaybackinfo](https://dev.emby.media/reference/RestAPI/MediaInfoService/postItemsByIdPlaybackinfo.html), [TV APK report](tv-apk-behavior-and-contracts.md#emby-wire-contracts-worth-re-specifying)).

Select among returned `MediaSourceInfo` records using their identifiers and declared support flags, then consume the server-provided `DirectStreamUrl` or `TranscodingUrl` and required HTTP headers. Use the media source's `RunTimeTicks`, not the containing item's runtime, and carry its `Id` together with the returned `PlaySessionId` through streaming and reporting ([Playback Guidelines](https://dev.emby.media/doc/restapi/Playback-Guidelines.html), [postItemsByIdPlaybackinfo](https://dev.emby.media/reference/RestAPI/MediaInfoService/postItemsByIdPlaybackinfo.html)).

Emby distinguishes direct play (client filesystem access), direct stream (original media streamed without encoding/remux), and transcoding. On a sandboxed mobile client, direct filesystem play is generally unavailable unless the source is genuinely client-accessible; ordinary server playback should use the negotiated HTTP URL. For progressive video, `/Videos/{Id}/stream.{Container}` uses `MediaSourceId` and `PlaySessionId`; `static=true` requests direct streaming, while transcoded seeking restarts with `StartTimeTicks`. HLS uses `/Videos/{Id}/master.m3u8` with `MediaSourceId` and `DeviceId` ([Playback Guidelines](https://dev.emby.media/doc/restapi/Playback-Guidelines.html), [Video Streaming](https://dev.emby.media/doc/restapi/Video-Streaming.html), [HTTP Live Streaming](https://dev.emby.media/doc/restapi/Http-Live-Streaming.html)).

## Audio and subtitle selection

Stream selection is by the `Index` values in the chosen media source's `MediaStreams`, not array position. Negotiation accepts `AudioStreamIndex` and `SubtitleStreamIndex`, and returned media sources declare `DefaultAudioStreamIndex` and `DefaultSubtitleStreamIndex`; changing tracks may therefore require a fresh playback-info request and a new negotiated URL rather than only a local player toggle ([postItemsByIdPlaybackinfo](https://dev.emby.media/reference/RestAPI/MediaInfoService/postItemsByIdPlaybackinfo.html), [getVideosByIdStreamByContainer](https://dev.emby.media/reference/RestAPI/VideoService/getVideosByIdStreamByContainer.html)).

Subtitle behavior follows the negotiated delivery method. `SubtitleProfile` can declare `Encode`, `Embed`, `External`, `Hls`, or `VideoSideData`; text streams marked `IsTextSubtitleStream` can be fetched as `/Videos/{Id}/{MediaSourceId}/Subtitles/{Index}/Stream.srt` or `.vtt`, with `StartPositionTicks` for an offset. Bitmap/embedded formats must not be treated as downloadable text, and burn-in is represented by selecting the subtitle index on the transcoded stream ([postItemsByIdPlaybackinfo](https://dev.emby.media/reference/RestAPI/MediaInfoService/postItemsByIdPlaybackinfo.html), [Subtitles](https://dev.emby.media/doc/restapi/Subtitles.html)).

## Session, progress, and user-state semantics

The session lifecycle is `POST /Sessions/Playing`, repeated `POST /Sessions/Playing/Progress`, and exactly one best-effort `POST /Sessions/Playing/Stopped`; all require user authentication. The reporting DTO carries the play-session, item, media-source, position/runtime ticks, pause/mute/seek state, play method, repeat mode, rate, and queue information, so those identifiers and tick units must remain consistent with the negotiated stream ([PlaystateService](https://dev.emby.media/reference/RestAPI/PlaystateService.html), [postSessionsPlayingProgress](https://dev.emby.media/reference/RestAPI/PlaystateService/postSessionsPlayingProgress.html)).

The client should report a start only after a play attempt is committed, progress periodically and on meaningful state changes (pause/resume/seek), and stop on completion, replacement, fatal error, or teardown. Those trigger choices are client policy inferred from the three official lifecycle operations; their cadence and retry/idempotency behavior are not specified by the public guide and must be verified with target-server tests ([PlaystateService](https://dev.emby.media/reference/RestAPI/PlaystateService.html)).

Played/unplayed and general user-data changes are distinct operations from playback check-ins: the service exposes played-item mutations and `POST /Users/{UserId}/Items/{ItemId}/UserData`. UI optimism must reconcile with the server-returned `UserData`, because it is the cross-client state authority ([PlaystateService](https://dev.emby.media/reference/RestAPI/PlaystateService.html), [Item Information](https://dev.emby.media/doc/restapi/Item-Information.html)).

## Compatibility contract and validation gates

The public documentation does not declare one Mobile Client minimum server version. At connection time record `/System/Info/Public` identity/version, then authenticate and use `/System/Info` for the canonical server `Id`; preserve unknown JSON fields and tolerate missing optional fields. Exact routes, enum values, defaults, and authorization requirements must be checked against the built-in API Browser of every supported server line because Emby identifies that interactive, running-server reference as the detailed API source ([SystemService reference](https://dev.emby.media/reference/RestAPI/SystemService.html), [getSystemInfo](https://dev.emby.media/reference/RestAPI/SystemService/getSystemInfo.html), [REST API documentation](https://dev.emby.media/doc/restapi/index.html)).

Before implementation is considered compatible, fixtures or integration tests must cover: base URLs with and without an `/emby` path; HTTP and valid HTTPS; 401 token revocation; multiple servers/users without credential crossover; unknown item types; reduced list DTOs; grouped latest episodes; missing and inherited images; multiple media sources; direct stream and transcode responses; no-compatible-stream errors; audio/subtitle index changes; external text versus burn-in subtitles; resume ticks; and start/progress/stop reporting. The APK report additionally requires testing `.strm` and server-specific source-priority behavior rather than treating its decompiled comparator or permissive TLS/cleartext settings as contract ([TV APK report](tv-apk-behavior-and-contracts.md#resources-and-reuse-boundary)).

Trust-all-certificate behavior and globally permissive cleartext are not Emby API requirements and must not be inherited. The supported transport policy, minimum Emby Server versions, reporting cadence, precise search endpoint, and initial device-profile matrix remain product/implementation decisions to lock with real-server fixtures; official documentation alone does not settle them.

## Planning decision

Specify one server-scoped Emby gateway with typed capabilities for connection/authentication, library queries, image requests, playback negotiation, stream selection, and session/user-state reporting. Its public contract should traffic in server/item/media-source/session identities and ticks, while URL construction, headers, DTO version tolerance, and device-profile generation stay encapsulated. Validate that gateway against each supported Emby Server line before freezing the Mobile Client spec.
